## 业务归档

```
请回顾当前会话，在 `archives/{{datetime}}.md` 中生成一份技术归档。要求如下：

1. **基本信息**：
   - **时间点**：精确到分钟。
   - **核心任务**：一句话描述当前处理的业务/功能模块。
2. **变更记录**：
   - **已完成工作**：列出本次会话实现的代码逻辑、UI 调整或配置更改。
   - **关键决策**：记录在实现过程中做出的重要技术选型或逻辑权衡（例如：为什么选了 A 而不是 B）。
3. **技术现状**：
   - **核心组件**：涉及到的关键接口、文件或依赖。
   - **环境变化**：是否新增了环境变量或依赖包（如 pnpm add）。
4. **待办/遗留**：
   - 记录尚未完成或需要后续优化的点（TODO）。

请保持语言简洁，以 Markdown 代码块形式输出。
```

## 收工检验

```
当前任务已经完成了，现在请你帮我进行最后的审查:

## 代码清洁与健壮性 (Cleanup)
- **冗余检查**：扫描本次修改，删除调试用的 console/log、未使用的变量及重复逻辑。
- **异常加固**：检查核心逻辑是否包含错误处理（Error Handling），确保非快乐路径下程序不会崩溃。
- **命名对齐**：检查新定义的变量/函数名是否与项目中既有的命名风格、业务术语一致。

## 架构合规性 (Audit)
- **职责分离**：检查是否有过度耦合的代码，建议是否需要将复杂的内联逻辑提取为独立的函数或组件。
- **依赖审计**：识别本次任务引入的新依赖或环境变量，并提醒我同步更新到 .env.example 或 README。

## 自动化归档 (Archive)
请在 `archives/$(date +%Y%m%d_%H%M).md` 中生成归档，包含：
- **Summary**: 简述本次 Session 解决了什么问题，实现了什么功能。
- **Changes**: 列出关键的文件变动及技术决策（为什么这么改）。
- **Breakpoints**: 如果我现在关闭会话，下一个 Session 应该从哪个具体的点接续？有哪些未完成的 TODO？

完成后请确认已执行上述操作，并展示归档文件的简要内容。
```
