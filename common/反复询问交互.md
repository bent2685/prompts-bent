### 业务澄清模式

```
当前用户提出的需求描述存在模糊空间。请在执行过程中严格遵守以下协议：

1. **业务反问**：利用询问工具（Ask User），针对需求中的模糊地带进行迭代式追问。重点挖掘：业务目的、用户操作链路、关键判断规则。
2. **拒绝冗余**：在此阶段，**严禁**生成任何技术设计文档或长篇调研报告。你的任务是『摸清业务逻辑』，而非展示技术细节。
3. **具象化确认**：在提问时，请尝试用『如果...那么...』的业务场景举例，向我确认你的理解是否正确。
4. **状态切换**：当你认为业务逻辑已 100% 闭环，请停止提问，并主动告知我：『业务逻辑已清晰，准备进入 Plan 模式』。

请直接开始第一波业务追问。
```

---

### 探索式业务对齐

```
请先 Deep Dive 相关代码库，分析现有业务逻辑对新需求的支持程度，然后执行：

1. **逻辑冲突探测**：找出新需求与现有代码逻辑可能冲突的地方，并针对这些点向我提问。
2. **最小化确认**：使用询问工具，确认我是否想复用现有逻辑，还是完全重构。提问要直击业务要害，避免空泛。
3. **严禁报告**：探索过程在后台静默完成，无需向我展示代码分析报告。你的输出必须全部是针对『业务实现』的询问。
4. **Plan 预热**：一旦业务逻辑摸清楚，请直接示意我切换到 Plan 模式，届时再输出具体的执行方案。

请先扫描代码，然后提出你的疑问。
```
