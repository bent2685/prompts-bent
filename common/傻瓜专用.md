### 意图读心模式

```
我接下来的描述可能非常模糊、非技术化，甚至只有几个关键词。请你执行以下逻辑：
- **脑补全貌**：根据当前项目上下文，推测我最可能想实现的完整业务场景。
- **自动拆解**：直接把这个模糊需求拆解成技术步骤，不需要问我『怎么做』。
- **只给是非题**：如果必须我确认，请直接告诉我：『我打算 A 这样做，如果你同意请扣 1，有异议请告诉我你的顾虑』。
我大概听不懂你说的技术术语，我只关心最后功能好不好用。
```

### 傻瓜式自愈指令

```
程序崩了/报错了。请闭嘴并执行：
1. **自主诊断**：自己去翻日志、看代码、查环境，别来问我。
2. **静默修复**：如果是拼写、路径、依赖、配置等低级错误，直接修好并告诉我『修好了』。
3. **人工确认**：如果是业务逻辑冲突，请用最通俗的语言解释冲突点，并给出你的『推荐一键修复包』。
**禁止向我展示堆栈追踪信息，我看不懂也不想看。**
```

### 代码人话翻译

```
请把这段代码/这个报错翻译成『三岁小孩能懂的业务逻辑』：
- 这个东西在干嘛？（功能）
- 谁坏了？（故障点）
- 我改哪里能让它变样？（操作点）
不要用任何编程术语，用生活中的类比（比如：水管漏了、快递丢了）来解释。
```

---

### 喂饭级决策建议

```
针对目前的 [问题/需求]，请直接给我一个**最优解**。
- 只有当我问『还有别的办法吗』时，你再提供备选方案。
- 告诉我这个解法的 **1 个好处** 和 **1 个坏处**（必须是小白能懂的，比如『运行变快了』或『以后改起来麻烦』）。
- 如果我没意见，请直接回复：『收到，正在一键执行』。
```
