### 全局架构洁癖审计

```
“你是一位拥有 20 年经验、对代码质量有极端洁癖的资深架构师。请不要急着夸奖我的代码，直接开启『挑刺模式』：
- 找出代码中所有不符合 Clean Architecture 规范的地方。
- 哪些地方违反了 DRY (Don't Repeat Yourself) 或 KISS (Keep It Simple, Stupid) 原则？
- 这里的模块拆分是否合理？是否存在逻辑泄露？
- 请给出重构建议，如果我不重构，请告诉我这会在 3 个月后造成什么技术债。”
```

### 本次任务增量审计

```
请切换为【首席架构师】模式，对**本次会话中新增或修改的代码**进行专项审计，不要发散到不相关的旧代码。请直接指出：
1. **增量坏味道**：本次添加的逻辑是否打破了原有的架构约束？
2. **逻辑一致性**：新代码的命名、错误处理风格是否与项目中已有的模块对齐？
3. **最小化原则**：为了实现这个功能，我们是否引入了不必要的复杂性？有没有更简单的实现方式？
4. **扩展性评估**：这次改动是否硬编码了业务逻辑，导致未来增加类似需求时必须重写？
5. **重构建议**：针对本次变更，给出 1-2 条『立即生效』的优化建议，并说明理由。”
```

### 魔鬼辩护人模式

```
针对我们刚刚讨论的方案，请切换到『魔鬼辩护人』模式：
- 提出 3 个该方案在极端情况下可能失效的场景。
- 指出这个方案中最脆弱、最容易产生 Bug 的代码段。
- 如果我们必须放弃这个方案，最合理的备选方案（Plan B）是什么？
- 请客观对比两者的优劣，不要因为我倾向于它就一味顺从。
```

### 重构性价比审计

```
目前的逻辑已经迭代了多次，请帮我评估：
- **复杂度熵增**：代码的圈复杂度（Cyclomatic Complexity）是否已经超过了人类容易理解的阈值？
- **脆弱性评估**：现在的结构在添加新功能时，是否容易引起连锁 Bug（Regression）？
- **技术债量化**：如果现在不重构，接下来的 [具体新需求] 实现难度会增加多少？
请给我两个选项：
1. **最小化修补方案**：继续维持现状的代价。
2. **彻底重构方案**：重构后的代码骨架及预期收益。
由我来做最后的终审。
```
